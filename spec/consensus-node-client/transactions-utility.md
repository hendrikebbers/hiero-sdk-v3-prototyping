# Utility Transactions API

This namespace groups utility transactions of the consensus node's *util* service — transactions
that do not belong to a specific entity service (accounts, tokens, topics, files, schedules) but
provide cross-cutting platform utilities. The first such transaction is `PrngTransaction`
(HIP-351), which asks the network for a verifiable pseudorandom value.

## Description

### `PrngTransaction` (HIP-351)

`PrngTransaction` requests a pseudorandom value derived from the network's running hash. The
randomness is generated by consensus — every node computes the same value from the same
deterministic seed — so the result is **verifiable** and not controllable by any single party
(including the submitter). The value is returned on the receipt, not as a query result, because it
is produced during transaction handling.

The payload is a single optional `range`:

- **`range` unset** → the receipt carries `prngBytes`: 384 bits (48 bytes) of pseudorandom data.
- **`range` set** (must be `> 0`) → the receipt carries `prngNumber`: a pseudorandom 32-bit integer
  uniformly distributed in the half-open interval `[0, range)`.

Exactly one of the two receipt fields is populated, mirroring which input mode was chosen; this is
expressed with `@@oneOf(prngNumber, prngBytes)` on `PrngReceipt`. The transaction changes no ledger
state beyond charging the fee — its sole observable effect is the value on the receipt.

## API Schema

```
namespace consensusnode.transactions.utility
requires {Receipt, Transaction} from consensusnode.transactions

// Requests a verifiable pseudorandom value from the network (HIP-351). The randomness is derived
// from the consensus running hash, so the result cannot be steered by the submitter.
//
//   - range unset      → the receipt carries prngBytes (384 bits / 48 bytes of randomness)
//   - range set (> 0)  → the receipt carries prngNumber, uniform in [0, range)
@@finalType
PrngTransaction extends Transaction<PrngReceipt> {
    @@immutable @@nullable @@min(1) range: int32   // unset → 384 random bits; set → a random int in [0, range)
}

// Carries the generated pseudorandom value. Exactly one field is set, matching the input mode of
// the PrngTransaction that produced it.
@@oneOf(prngNumber, prngBytes)
@@finalType
PrngReceipt extends Receipt {
    @@immutable @@nullable prngNumber: int32   // set when range was provided; uniform in [0, range)
    @@immutable @@nullable prngBytes: bytes    // set when range was unset; 384 bits (48 bytes) of randomness
}
```

## Examples

### Request a random number in a bounded range

A lottery picks a winning ticket out of 10 000 sold. Supplying `range` makes the network return a
number in `[0, 10000)` directly.

```
HieroClient client = ...;

PrngReceipt receipt = new PrngTransaction()
    .range(10000)
    .signWithOperatorAndSubmit(client)
    .queryReceipt();

int32 winningTicket = receipt.prngNumber;   // 0 .. 9999
```

### Request raw random bytes

Without `range`, the receipt yields 48 bytes of randomness — e.g. to seed a client-side generator
or derive a nonce.

```
PrngReceipt receipt = new PrngTransaction()
    .signWithOperatorAndSubmit(client)
    .queryReceipt();

bytes seed = receipt.prngBytes;   // 48 bytes
```

## Questions & Comments

- **`prngNumber` is a signed `int32`, but the value is always non-negative.** HAPI returns the
  bounded result as an `int32` in `[0, range)`; `range` itself is a positive `int32`, so the result
  never overflows into the sign bit for any `range`. The field is left as `int32` (rather than an
  unsigned type the meta-language does not have) to match HAPI exactly; the `[0, range)` contract is
  documented on the field.

- **No client-side validation that the network supports HIP-351.** A ledger whose consensus node
  does not implement the util service rejects the transaction with a `TransactionStatus` on the
  receipt, consistent with how every other service-specific transaction surfaces unsupported
  operations (see [`transactions-schedule.md`](transactions-schedule.md) for the same reasoning
  about schedulable inner transactions).

- **Why a transaction and not a query.** The value is produced *during* consensus handling and
  recorded on the receipt; it cannot be read back without changing the network's running-hash state.
  Modelling it as a `Transaction<PrngReceipt>` keeps it on the existing receipt-per-transaction
  pattern (see [`missing-features.md`](../../missing-features.md) §3.5) rather than inventing a
  side-effecting query.
